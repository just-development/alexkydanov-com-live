# JUS-67 - Cloudflare Tunnel ingress for alexkydanov.com office origin

## Scope
- Domain: `alexkydanov.com` (+ `www.alexkydanov.com`)
- Origin host: `web-projects` (`192.168.35.95`)
- Objective: route public traffic through Cloudflare Tunnel (outbound from office), replacing direct public-IP dependency.

## Prerequisites
1. Cloudflare API token with both:
- `Account Cloudflare Tunnel:Edit`
- `Zone DNS:Edit` for `alexkydanov.com`
2. SSH access to `web-projects` using `cto@100.113.84.79` with key `~/.ssh/cto_web_projects`.
3. Limited NOPASSWD sudo allowing file/service management (`cp`, `chmod`, `mkdir`, `systemctl`, etc.).
4. Local web service on `web-projects` reachable via `http://127.0.0.1:80`.

## Cloudflare account-side setup
Run from repo root:

```bash
CLOUDFLARE_API_TOKEN=... ./scripts/cloudflare/setup_alexkydanov_tunnel.sh
```

This does:
1. Backup current DNS to `ops/cloudflare/alexkydanov.com/dns-backup-before-tunnel.json`
2. Create/find tunnel `alexkydanov-office-origin`
3. Replace apex GitHub Pages `A` records with proxied tunnel CNAME
4. Point `www` CNAME to same tunnel target
5. Write metadata to `ops/cloudflare/alexkydanov.com/tunnel-bootstrap-output.json`

## Origin-side setup on web-projects
On `web-projects`:

```bash
# install cloudflared without apt/dpkg (fits limited sudo command set)
curl -fsSL https://github.com/cloudflare/cloudflared/releases/latest/download/cloudflared-linux-amd64 -o ~/cloudflared
chmod +x ~/cloudflared
sudo cp ~/cloudflared /usr/local/bin/cloudflared
sudo chmod 755 /usr/local/bin/cloudflared

# create config dir
sudo mkdir -p /etc/cloudflared
sudo chmod 750 /etc/cloudflared
```

Copy tunnel credentials JSON generated by Cloudflare to:
- `/etc/cloudflared/<TUNNEL_ID>.json`

Then deploy files from this repo:
- `ops/cloudflare/alexkydanov.com/cloudflared-config.yml` -> `/etc/cloudflared/config.yml`
- `ops/cloudflare/alexkydanov.com/cloudflared.service` -> `/etc/systemd/system/cloudflared.service`

Replace `TUNNEL_ID` in config with actual tunnel id.
Current service template runs as `cto` to stay inside the approved privilege envelope.

Start service:

```bash
sudo systemctl daemon-reload
sudo systemctl enable cloudflared
sudo systemctl restart cloudflared
sudo systemctl status --no-pager cloudflared
```

## Canonical host behavior
Target behavior: apex canonical (`alexkydanov.com`), `www` redirects to apex.

Recommended edge rule in Cloudflare:
- If Hostname equals `www.alexkydanov.com`, redirect `301` to `https://alexkydanov.com${uri}`.

Until that rule is applied, both hosts are served by tunnel.

## Validation
From an external network:

```bash
curl -I https://alexkydanov.com
curl -I https://www.alexkydanov.com
```

Expected:
- `alexkydanov.com` returns `200` or app-specific success.
- `www.alexkydanov.com` returns `301` to apex after redirect rule is enabled.

## TLS posture notes
- Client-facing TLS terminates at Cloudflare edge by default.
- Origin between Cloudflare Tunnel and local service is inside tunnel transport.
- If origin HTTPS is needed later, switch ingress service to `https://127.0.0.1:443` and configure origin certificate trust in `cloudflared`.

## Rollback
To restore pre-cutover DNS:

```bash
CLOUDFLARE_API_TOKEN=... ./scripts/cloudflare/rollback_alexkydanov_tunnel_dns.sh
```

This restores apex/www records from backup JSON.
